## 1. 한눈에 보기

meetcute는 **소수의 관리자가 함께 쓰는 소개팅 주선 관리 도구**입니다. 매물(소개해줄 사람들), 만남 기록, 매칭 상태를 한 곳에서 관리합니다.

| 메뉴 | 경로 | 권한 | 용도 |
|---|---|---|---|
| 가입 | `/auth/register` | 누구나 | 계정 만들기 |
| 로그인 | `/auth/login` | 누구나 | 로그인 |
| 비밀번호 재설정 | `/auth/forgot` | 누구나 | 텔레그램으로 1회용 링크 받기 |
| 대시보드 | `/` | 관리자 | 카운트, 진행 중 매칭, 최근 활동 |
| 매물 목록 | `/persons` | 관리자 | 검색/필터, 상태 뱃지 확인 |
| 새 매물 | `/persons/new` | 관리자 | 등록 |
| 매물 상세 | `/persons/{id}` | 관리자 | 정보·사진·만남 이력 |
| 만남 기록 목록 | `/encounters` | 관리자 | 결과별 필터 |
| 새 만남 | `/encounters/new` | 관리자 | 두 매물 골라 기록 |
| 호환성 체크 | `/compatibility` | 관리자 | 두 매물 빠른 비교 |
| 마담뚜 대화 | `/chat` | 관리자 | 마담뚜끼리 임시 대화방 (72h 휘발) |
| 유저 관리 | `/users` | 관리자 | 가입자 권한 토글/삭제 |
| 매뉴얼 | `/manual` | 누구나 | 이 문서 |

## 0. 인증 & 권한

### 0.-2 외부 노출 (cloudflared 터널) 🌐

같은 와이파이 아닌 사람도 접속하게 하려면 cloudflared 터널이 가장 깔끔합니다.

**한 번 셋업:**
```bash
brew install cloudflared
```

**빠른 모드 (랜덤 URL, 즉시):**
```bash
# 별도 터미널에서 (dev.sh 가 8765에 떠있는 상태로)
cloudflared tunnel --url http://localhost:8765
```
콘솔에 `https://<random>.trycloudflare.com` URL 뜨면 친구한테 공유. 끄면 URL 사라짐.

**안정 모드 (영구 URL, cloudflare 계정 + 도메인 필요):**
```bash
cloudflared tunnel login                                        # 브라우저 인증
cloudflared tunnel create meetcute
cloudflared tunnel route dns meetcute meetcute.yourdomain.com
cloudflared tunnel run meetcute
```

**🔄 URL 변경 자동 알림:**
`./tunnel.sh` 가 cloudflared 출력에서 새 URL 을 파싱해 `.public_url` 파일에 기록합니다. 앱은 매분 이 파일을 확인하고, **URL 이 변경되면 telegram_chat_id 가 등록된 모든 admin 에게 새 URL 을 텔레그램으로 자동 푸시**해줍니다. 노트북 재부팅이나 cloudflared 재시작 시 quick 모드 URL 이 바뀌어도 자동으로 알림이 가니, 다른 admin 들이 새 URL 을 받아서 홈 화면에 다시 추가하기 편함.

**🔒 외부 노출 시 보안 강화 (필수):**
`.env` 에 추가:
```
MEETCUTE_PUBLIC=on
MEETCUTE_AUTH=on   # PUBLIC=on 이면 자동 강제됨
```
- 세션 쿠키가 HTTPS-only 로 전환 (트래픽 가로채기 방지)
- 상단에 분홍색 "🌐 공개 모드" 배너 표시
- 인증 꺼져있어도 자동으로 켜짐

dev.sh 재시작하면 적용됩니다.

### 0.-1 자리 비울 때 자동 업데이트: `./dev.sh`

평소엔 `uvicorn ...` 으로 직접 띄워도 되지만, **외부에서 코드가 푸시될 가능성이 있고 본인이 노트북을 직접 만지기 어려운 상황**이면 다음 스크립트로 띄우세요:

```bash
./dev.sh
```

이 스크립트는 백그라운드에서 15초마다 `git fetch` 해서:
- 새 커밋이 들어오면 자동 `git pull` → uvicorn이 알아서 리로드
- `pyproject.toml` 변경(=새 라이브러리)도 감지 → 자동 `pip install` 후 강제 재시작
- 서버가 죽으면 자동 재기동
- 충돌 발생 시 안전하게 멈추고 로그만 남김

옵션은 환경변수로:
```bash
PORT=8001 ./dev.sh                    # 포트 변경
HOST=0.0.0.0 ./dev.sh                 # 같은 와이파이 기기들도 접근 가능
POLL_INTERVAL=5 ./dev.sh              # 5초마다 깃 확인 (기본 15초)
MEETCUTE_AUTH=on ./dev.sh             # 인증 켜기
```

> 주의: 본인이 로컬에서 만든 커밋이 있는데 외부 푸시도 들어왔다면 fast-forward 실패로 `dev.sh`가 멈춰요. 평소엔 외부 푸시만 받는 모드로 쓰시고, 본인이 코드 만질 땐 일반 `uvicorn` 으로 띄우는 게 깔끔합니다.

### 0.0.0.5 백업 (Dropbox / iCloud / 외장하드) 💾

데이터는 두 폴더에만 있습니다 — 이 둘만 어딘가에 복사해두면 끝.
- DB 파일: `meetcute/data/meetcute.db`
- 사진: `meetcute/uploads/`

`/settings` 페이지의 "💾 데이터 / 시스템 상태" 카드에서 정확한 경로 확인 가능.

#### 방법 A — 심볼릭 링크 (Dropbox 가 알아서 동기화) ⭐ 자동
**한 번만 셋업하면 영원히 자동.** 단 본인 노트북 한 대에서만 사용 가정.

```bash
# 1) 서버 꺼두기
Ctrl+C   # dev.sh 종료

cd ~/easyseop/meetcute

# 2) data/uploads 를 Dropbox 안으로 이동
mkdir -p ~/Dropbox/meetcute-backup
mv data ~/Dropbox/meetcute-backup/data
mv uploads ~/Dropbox/meetcute-backup/uploads

# 3) 원래 경로엔 심볼릭 링크
ln -s ~/Dropbox/meetcute-backup/data data
ln -s ~/Dropbox/meetcute-backup/uploads uploads

# 4) 서버 재시작
./dev.sh
```

이후 매물 등록할 때마다 Dropbox 가 자동 동기화. iCloud 도 같은 방식 (`~/Library/Mobile\ Documents/com~apple~CloudDocs/` 가 iCloud 경로).

> ⚠️ 두 대 이상의 노트북에서 동시에 같은 Dropbox 의 SQLite DB 에 접속하면 데이터 깨질 수 있음 (SQLite 는 단일 writer 가정). **메인 노트북 한 대에서만** 쓰세요.

#### 방법 B — 주기적 rsync 백업 (안전, 일 1회)
SQLite 의 동시쓰기 위험 피하고 싶으면 그냥 정기 복사:

```bash
# 수동
rsync -av ~/easyseop/meetcute/data ~/easyseop/meetcute/uploads ~/Dropbox/meetcute-backup/

# 자동 (매일 새벽 2시) — crontab -e 열어서 한 줄 추가
0 2 * * * rsync -a ~/easyseop/meetcute/data ~/easyseop/meetcute/uploads ~/Dropbox/meetcute-backup/ 2>&1 | logger -t meetcute-backup
```

#### 방법 C — 그냥 가끔 손으로
번거롭지만 가장 안전. 매물 많이 등록한 날 저녁에 한 번씩:
```bash
cp -R ~/easyseop/meetcute/data ~/Dropbox/meetcute-backup/data-$(date +%Y%m%d)
cp -R ~/easyseop/meetcute/uploads ~/Dropbox/meetcute-backup/uploads-$(date +%Y%m%d)
```
(날짜 박혀서 여러 시점 보관됨)

#### 복구
백업한 폴더를 원위치로 덮어쓰기:
```bash
Ctrl+C   # 서버 끄고
cp -R ~/Dropbox/meetcute-backup/data/* ~/easyseop/meetcute/data/
cp -R ~/Dropbox/meetcute-backup/uploads/* ~/easyseop/meetcute/uploads/
./dev.sh
```

### 0.0.0 데이터베이스 (어디에 저장됨?)

- **기본은 SQLite** — `meetcute/data/meetcute.db` 단일 파일에 자동 생성. 별도 설치 X.
- 스키마(테이블)는 첫 부팅 시 자동 생성

**MySQL/PostgreSQL 쓸 거면** 환경변수 `MEETCUTE_DB_URL`:
```bash
export MEETCUTE_DB_URL="mysql+pymysql://USER:PASS@HOST:3306/meetcute?charset=utf8mb4"
```
MySQL인 경우 `meetcute` 데이터베이스가 없으면 자동 생성됩니다 (`CREATE DATABASE IF NOT EXISTS`).

**시드 데이터 (가짜 매물 8명 + 만남 + 이벤트):**
```bash
python -m app.seed              # 데이터 없을 때만
python -m app.seed --force      # 다 지우고 다시
```

**백업:**
- SQLite: `meetcute/data/meetcute.db` 파일 복사
- MySQL: `mysqldump -u root meetcute > backup.sql`
- 사진은 어느 DB든 `meetcute/uploads/` 폴더 (이 폴더도 함께 복사)

### 0.0 인증 켜기/끄기 토글 ⭐

기본은 **꺼져 있습니다.** 혼자 로컬에서 쓸 때는 그냥 띄우면 바로 모든 페이지 접근 가능 — `(local)` 관리자로 동작.

```bash
# 인증 꺼짐 (기본). 로그인/가입 화면 없음.
uvicorn app.main:app --reload

# 인증 켜기. 다시 로그인/가입/유저 관리 활성화.
MEETCUTE_AUTH=on uvicorn app.main:app --reload
```

켜진 상태에선 §0.1 ~ §0.5 그대로 작동합니다.
꺼진 상태에선 화면 상단에 🔓 노란 띠로 표시되며, 매물 정보 변경 이력의 편집자는 `(local)` 로 기록됩니다.

> 💡 인증을 다시 켤 때 이미 DB에 가입된 유저가 없으면, 첫 가입자가 자동 관리자가 됩니다 (§0.1).

### 0.1 첫 실행 (부트스트랩)
1. 앱을 처음 띄우면 DB에 유저가 한 명도 없어요
2. `/auth/register` 에서 첫 계정을 가입 → **자동으로 관리자 권한 부여** 🎉
3. 이후 가입자는 기본적으로 권한 없음 (대기 상태)

### 0.2 새 관리자 추가하기
1. 새 사람에게 `/auth/register` 링크를 공유 → 이메일/비밀번호로 가입하라고 안내
2. 가입 후 그 사람은 "승인 대기 중" 페이지만 보임
3. 기존 관리자가 `/users` 에 가서 해당 유저 옆 **관리자로** 버튼 클릭
4. 그 사람이 새로고침하면 즉시 매물 관리 기능 사용 가능

### 0.3 권한 강등/삭제
- `/users` 에서 **강등** 버튼으로 일반 유저로 변경 (계정은 유지)
- **삭제** 버튼으로 계정 자체를 제거
- ⚠️ **마지막 관리자는 강등/삭제 불가** (시스템에 관리자 0명 방지)

### 0.4 비밀번호
- 최소 8자 (변경/재설정 시엔 10자 이상), bcrypt로 해싱되어 Fernet 으로 한 번 더 암호화 저장
- **비밀번호 변경**: 로그인 상태에서 `/settings` 에서 현재 비번 + 새 비번
- **비밀번호 재설정**: `/auth/login` 페이지의 "비밀번호를 잊으셨나요?" → 이메일 입력 → 텔레그램으로 30분 만료 1회용 링크 발송 → 새 비번 입력 → 자동 로그인
  - 텔레그램 chat_id 가 등록된 마담뚜만 가능 (`/settings` 에서 등록)
  - 미등록 마담뚜는 책임자에게 문의

### 0.5 운영 시 주의
- 환경변수 **`MEETCUTE_SECRET`** 을 강한 임의값으로 지정 (세션 쿠키 위조 방지)
  - 미설정 시 dev 기본값을 쓰며, 콘솔에 경고 출력됨
- HTTPS 환경에서는 `app/main.py`의 `https_only=True` 로 변경
- CSRF 보호는 아직 없음 (외부 노출 시 추가 필요)

## 2. 매물(=소개해줄 사람) 등록

### 2.1 등록 흐름
1. 우측 상단 **+ 새 매물** 또는 `/persons/new` 진입
2. **성별** 선택 (등록 후 변경 불가 — `public_id` 발급 기준)
3. 필수: 나이, 키, 거주지, 직장
4. 선택: 별칭(본인만 보는 메모용), 이상형, 주선자 메모
5. 사진 추가 (최대 5장, jpg/png/webp/heic)
6. **등록** 클릭 → 자동으로 `M-001`, `F-001`, `X-001` 식 ID 발급

### 2.2 ID 규칙
- **M-001** = 남성 1번째, **F-001** = 여성 1번째, **X-001** = 기타 1번째
- 같은 성별 내 가장 큰 번호 + 1
- 매물을 삭제해도 번호는 재사용되지 않습니다 (이력상 혼동 방지)

### 2.3 별칭 vs 실명
- **실명은 절대 저장하지 마세요.** 노출 시 신뢰 문제로 직결됩니다.
- 별칭은 본인만 보는 메모용 (예: "동아리 후배", "회사 김대리 친구")

### 2.4 사진 업로드
- 최대 5장, 첫 번째 사진이 목록 카드 썸네일이 됨
- 매물 수정 페이지에서 개별 사진 삭제 가능 (호버 시 × 버튼)
- **일괄 다운로드**: 상세 페이지의 `⬇️ 전체 다운로드 (zip)` 버튼 → `{public_id}_photos.zip` 으로 원본 받아요 (썸네일 제외)

## 3. 매물 상태 (자동 계산 + 수동 오버라이드)

매물의 상태는 기본적으로 만남 기록(Encounter) 들의 결과를 보고 **자동** 결정되지만, 담당자가 매물 상세에서 **수동 지정**할 수도 있어요.

### 3.1 자동 계산 규칙

| 상태 | 뱃지 | 판단 기준 |
|---|---|---|
| 소개 가능 | 🟢 초록 | 만남 기록이 없거나 모두 종료됨 |
| 진행 중 | 🟡 노랑 | PENDING/CONTINUING 만남 1개 이상 |
| 매칭됨 | 💗 분홍 | MATCHED 만남이 1개라도 있음 |

> **주의:** "동시 진행 허용" 정책이라 한 사람이 여러 명과 진행 중일 수 있습니다.
> "MATCHED"는 한 번 찍히면 다른 만남 결과와 무관하게 매칭됨으로 표시됩니다.

### 3.2 수동 변경 (오버라이드) 🔧

매물 상세 페이지에서 상태 뱃지 옆 **`상태 변경`** 링크를 누르면 모달이 뜨고, 다음 5가지 중 골라요:

| 옵션 | 의미 |
|---|---|
| 🤖 자동 | 기본값. 만남 기록에 따라 자동 계산 (위 표) |
| 🟢 소개 가능 | 만남 진행 중이어도 강제로 '소개 가능' 으로 |
| 🟡 진행 중 | 강제로 '진행 중' |
| 💞 매칭됨 | 강제로 '매칭됨' |
| ⛔ 불가 | 매물이 휴식기/본인 사정으로 잠시 소개 불가 (자동으론 안 나오는 상태) |

- 수동 지정된 매물은 뱃지에 **🔧** 표시가 붙어요 — "자동이 아니다" 신호.
- 다시 자동으로 돌리려면 모달에서 **🤖 자동** 선택 → 저장.
- **권한**: 그 매물 담당 마담뚜 본인 또는 책임자만 변경 가능.
- 변경 내역은 활동 로그(`/activity`) 에 `person.status_override` 로 기록됩니다.

## 4. 만남 기록 (Encounter)

### 4.1 새 만남 기록
1. `/encounters/new` 또는 매물 상세에서 **+ 새 만남**
2. 매물 A, B를 각각 골라요 (같은 매물은 안 됨)
3. 만난 날짜, 결과, 메모 입력
4. **기록** → 편집 페이지로 이동

> 매물 상세에서 **+ 새 만남**을 누르면 그 매물이 A로 자동 선택돼요.
> **호환성 체크** 화면에서 "이 둘로 만남 기록 만들기"를 누르면 둘 다 자동 채워져요.

### 4.2 결과(outcome) 종류
| 값 | 의미 | 상태 영향 |
|---|---|---|
| PENDING | 만남 잡혔거나 결과 미정 | → 진행 중 |
| CONTINUING | 계속 만나는 중 | → 진행 중 |
| MATCHED | 매칭 성공 (사귀기로) | → 매칭됨 |
| ENDED_A | A가 거절 | → 영향 없음 (이력만) |
| ENDED_B | B가 거절 | → 영향 없음 |
| MUTUAL_END | 양쪽 다 노 | → 영향 없음 |

### 4.3 결과 업데이트
- 만남이 진전될 때마다 PENDING → CONTINUING → MATCHED 식으로 갱신
- `/encounters/{id}` 또는 매물 상세 타임라인의 **편집** 링크
- 메모는 자유롭게 누적해서 적어두세요 (타임라인에 노출됨)

### 4.4 결과 변경 이력 (📜 자동 기록)

만남 한 건의 outcome이 바뀔 때마다 **자동으로 이력이 쌓입니다.** 편집 페이지 하단 "📜 결과 변경 이력" 섹션에서 확인:

- 언제 어떤 결과로 바뀌었는지 시간순 타임라인
- 결과를 바꿀 때 "결과 변경 사유" 입력란에 짧게 메모하면 그게 같이 기록됨 (선택 사항)
- 누가 바꿨는지 이메일도 표시 (인증 끈 모드면 `(local)`)
- "전체 메모" 만 수정하고 결과가 그대로면 이벤트는 안 생김 (소음 방지)

> 💡 예시 흐름:
> 1. 새 만남 등록 → 자동으로 `PENDING / 최초 등록` 이벤트 1건
> 2. 다음날 편집해서 `CONTINUING` + "1차 만남 잘 됐다고 함" 입력 → 이벤트 2번째
> 3. 한 달 뒤 `MATCHED` + "사귀기로 했음!" → 이벤트 3번째
>
> 이렇게 쌓아두면 나중에 "이 매칭은 한 달 만에 됐네", "저건 3개월 PENDING 끌다가 결국 ENDED 됐네" 같은 분석이 그대로 됩니다.

## 4.5 매물 활동 추적 (이력)

매물 상세에 자동으로 표시됩니다:

- **총 만남 / 매칭 / 진행 중 / 마지막 활동** 카드 4종
- **만남 이력 타임라인** — 누구와 언제, 결과 메모
- **📝 정보 변경 이력** — 매물 정보를 수정할 때마다 자동 스냅샷, 어떤 필드가 어떻게 바뀌었는지 색깔로 표시 (빨간 줄=이전, 초록=현재)
  - 추적 필드: 나이, 거주지, 직장, 키, 이상형, 메모, 별칭
  - **누가 변경했는지 이메일도 같이 기록됨**
  - 사진만 추가/삭제한 경우는 변경 이력에 안 남음

### 매물 목록에서 활동으로 필터링
`/persons` 페이지 상단 필터:
- **활동 무관 / 진행 중 있음 / 30일+ 미활동 / 한 번도 만남 X**
- **정렬: 등록순 / 최근 활동순 / 오래 잠자는순**

> 💡 **TIP** — "오래 잠자는" 매물은 대시보드 상단에서 자동으로 노출되니
> 거기서 바로 클릭해 누구한테 소개해줄지 검토하면 됩니다.

### 4.6 다른 admin 매물한테 소개 요청 보내기 📨 (AUTH=on 전용)

같이 쓰는 admin이 2명+ 일 때:

1. 다른 admin이 운영하는 매물(=다른 owner)을 매물 목록에서 본다 → 필터 "다른 admin" 또는 "전체"
2. 그 매물 상세 페이지에 **📨 소개 요청** 버튼이 떠요
3. 버튼 누르면 → "내 매물 중에서 누구로 소개?" 드롭다운 + 메모 입력 → 보내기
4. 상대 admin이 텔레그램 알림 받고 (등록되어 있으면) → `/requests` 열어서 확인
5. 상대가 매물에 IRL로 물어본 뒤 **수락** 또는 **거절** 응답
   - **수락**: 만남 기록(Encounter)이 PENDING 상태로 자동 생성됨. 양쪽 매물 타임라인에 즉시 노출.
   - **거절**: 사유 메모와 함께 보낸이에게 알림

요청 상태:
| 상태 | 의미 |
|---|---|
| PENDING | 받은이가 아직 응답 안 함 |
| ACCEPTED | OK → 만남 기록 자동 생성됨 |
| DECLINED | 거절 |
| WITHDRAWN | 보낸이가 취소 (응답 전에만 가능) |

`/requests` 페이지에서 본인의 **받은 요청** / **보낸 요청** 두 섹션을 같이 봅니다.

### 4.8 텔레그램으로 매물 등록 🤖

봇이 보내는 알림만 받는 게 아니라, **텔레그램 채팅으로 새 매물을 등록**할 수도 있습니다 (모바일에서 가장 편함).

**준비:**
1. 웹 `/settings` 에서 본인 chat_id 등록되어 있어야 함 (4.7 참조)
2. 그 사람이 관리자 권한 보유

**명령어:**
| 명령 | 동작 |
|---|---|
| `/start` | 본인 chat_id 확인 + 연결된 계정 표시 |
| `/help` | 명령어 목록 |
| `/me` | 연결된 계정 정보 (닉네임/권한) |
| `/register` | 새 매물 등록 9단계 시작 |
| `/cancel` | 진행 중인 등록 취소 |
| `/done` | (사진 단계에서) 등록 마무리 |

**등록 흐름 (`/form` 또는 `/register`):**

1. `/form` 입력 → 봇이 양식 보냄:
   ```
   성별:
   이름:
   사는곳:
   직장:
   나이:
   키:
   (선택)
   이상형:
   메모:
   ```
2. 양식 그대로 복사 → 빈 칸 채워서 → 한 번에 답장
   ```
   성별: F
   이름: 민지
   사는곳: 서울 마포구
   직장: ○○회사 마케팅팀
   나이: 28
   키: 165
   ```
3. 봇이 검증 → 사진 보내기 단계로
4. 사진 1~5장 보내고 `/done` → 매물 자동 등록 (EXIF 회전 적용)

검증 실패 시 (`성별 X`, `나이 범위 X` 등) 어떤 줄이 잘못됐는지 알려주니 수정해서 다시 보내면 됨. `/cancel` 로 언제든 취소.

> 💡 **이상형 / 주선자 메모는 양식에서 빠뜨려도 됩니다.** 등록 후 웹 `/persons/{id}/edit` 에서 추가 가능 — 모바일 등록은 빠르게, 자세한 정보는 나중에.

**제약:**
- 1:1 채팅(private)에서만 작동. 그룹/채널 메시지는 무시.
- 봇은 알림 + 등록만 처리. "수락/거절" 등 다른 액션은 웹 `/requests` 에서.
- 진행 중인 등록 상태는 in-memory — 서버 재시작 시 진행 중 세션 손실 (이미 commit 된 매물은 안전).

### 4.7 텔레그램 알림 설정 (선택)

`/settings` 에서 본인 텔레그램 chat_id 등록 → 위의 소개 요청 이벤트(생성/수락/거절/취소)마다 텔레그램으로 알림이 옵니다.

**서버 준비 (한 번):**
1. 텔레그램 [@BotFather](https://t.me/BotFather) 한테 `/newbot` → 봇 토큰 받기
2. 서버 띄울 때 환경변수로:
   ```bash
   MEETCUTE_TELEGRAM_BOT_TOKEN="123456:abc..." ./dev.sh
   ```

**각 admin 준비 (한 번씩):**
1. 텔레그램에서 본인의 chat_id 알아내기: [@userinfobot](https://t.me/userinfobot) 한테 `/start`
2. 본 서비스 봇 (관리자가 알려준 봇) 에도 `/start` 한 번 → 안 보내면 봇이 본인한테 메시지 못 보냄
3. `/settings` 페이지에서 chat_id 입력 → 저장 → **🧪 테스트 메시지 보내기** 로 검증

토큰 미설정/chat_id 미등록 시엔 알림은 그냥 안 가고 앱은 정상 동작합니다.

### 4.9 마담뚜 임시 대화방 💬 (AUTH=on 전용)

마담뚜끼리 매칭 건을 조율할 1:1 대화방. 상단 메뉴 **💬 대화** (`/chat`).

**흐름:**
1. `/chat` 에서 상대 마담뚜를 고르고 (선택적으로 주제 입력) **대화 시작**
   - 같은 상대와 이미 살아있는 방이 있으면 새로 안 만들고 그 방으로 이동
   - 또는 **다른 마담뚜 소유 매물 상세 페이지**의 **`💬 담당 마담뚜와 대화`** 버튼 → 그 매물 담당자와 바로 방 생성/이동 (주제 `{매물ID} 건` 자동 입력)
2. 방에 들어가 카톡처럼 메시지 주고받기 — 웹에서 2~3초마다 자동 갱신(실시간)
3. 상대가 앱을 닫고 있어도 **텔레그램으로 "새 메시지" 알림** (직전 메시지가 5분 이내면 도배방지로 스킵)

**시각 · 읽음 표시:**
- 메시지마다 보낸 시각(오전/오후 H:MM, 한국시간)이 표시됩니다.
- 내가 보낸 메시지엔 **읽음 / 안읽음** 이 떠요 — 상대가 방을 열면(또는 보고 있으면) 자동으로 '읽음'으로 바뀝니다.

**수명 (휘발성):**
- **마지막 메시지로부터 72시간** 지나면 방·메시지가 **자동 삭제**됩니다 (sliding — 대화가 활발하면 안 사라짐).
- 환경변수 `MEETCUTE_CHAT_TTL_HOURS` 로 조절 가능 (기본 72).
- 방 안의 **🗑️ 방 닫기** 버튼으로 즉시 삭제도 가능.

**권한:**
- 두 참여자만 입장 + 전송.
- **책임자(👑)는 모든 방 열람 가능** (단, 전송은 참여자만 — 책임자는 읽기 전용).
- 제3의 마담뚜는 URL 알아도 접근 불가(403).

**프라이버시:** 메시지 본문은 DB에 **암호화 저장**되고, 72시간 후 완전 삭제됩니다. 매물 실명·전화는 여기서도 DB에 남기지 마세요(정책 동일).

## 7. 삭제 정책 (중요)

### 매물 삭제
- **하드 삭제**: DB row + 사진 파일 + **정보 변경 이력**까지 디스크에서 영구 제거
- 그 매물이 등장한 **만남 기록은 보존됨**
  - FK는 NULL이 되고 `public_id (deleted)` 스냅샷이 남음
  - 타임라인/목록에 취소선으로 표시됨
- ⚠️ 한 번 삭제하면 복구 불가. 보관할 거면 삭제 대신 그냥 두세요.

### 만남 기록 삭제
- 잘못 입력했을 때만 사용 (편집 화면 하단)
- 결과가 ENDED여도 **이력으로 남기는 게 원칙**
- 삭제 시 그 만남의 **결과 변경 이력(EncounterEvent)도 함께 삭제**됨
